问题

传统数据平台开发的六大困境

企业构建数据仓库不缺工具，缺的是把需求、设计、开发、验证、交付、运维串起来的系统工程能力

DILEMMA 01

需求理解断层

业务方说"我要看开票趋势"，技术方听不懂字段含义。需求文档、报表原型、KPI 散落在邮件和 Excel 中，缺乏从需求到模型的结构化映射规范。

DILEMMA 02

数仓设计复杂

ODS 放什么、DWD 怎么拆、DWS 聚合到哪个粒度、DIM 用哪种 SCD 策略——每层设计都要反复推敲。没有标准化的分层规范和引用约束，质量全靠个人经验。

DILEMMA 03

SQL 编写低效重复

数百张表，每张手写类型转换、NULL 处理、去重逻辑、审计字段。换一个源系统，所有 SQL 推倒重来。80% 是模板代码，却消耗了 80% 的时间。

DILEMMA 04

数据验证靠肉眼

数据从 Hive 迁到 OceanBase 后，行数对不对？金额合计数一不一致？主键是否唯一？依赖人工抽样对比，几百张表根本验不过来，上线后才发现差异。

DILEMMA 05

看板搭建耗时

FineReport 数据集一个一个配，SQL 一条一条粘，筛选参数手动绑定。做个报表从原型到上线 3 天起步，底层表结构一改，上面全重做。

DILEMMA 06

部署运维复杂

ETL 链几十个节点，先后顺序、失败重试、跑完验证全在运维脑子里。没有版本化的调度配置，没有自动化的 CI/CD 门禁，换人即瘫痪。

解决方案

AI + 数据平台 · 双引擎协同

AI 负责理解需求、设计模型、生成代码、判断差异——替代人的脑力劳动。数据平台负责类型转换、审计追踪、血缘计算、跨库对比、调度编排——替代人的手工操作。二者分工协作，形成完整的数仓建设流水线。

🧠 AI 引擎

8 个 Agent 组成认知流水线——
· 理解业务需求，设计五层模型
· 从字段清单自动生成标准化 SQL
· 判断跨库差异是否可接受
· 将 project.yml 渲染为执行上下文
→ 替代 SQL 编写、需求理解、设计推敲

⚙️ 数据平台引擎

8 大引擎 提供工程化能力——
· 适配器模式统一处理跨源类型转换
· 7 列审计字段自动注入全链路模型
· dbt DAG 引擎按拓扑顺序编排执行
· 四维自动验证 + CI/CD + 定时调度
→ 替代手工验证、手动调度、重复配置

00
AGENT-00
规范制定
负责：制定编码规范、命名标准、测试模板，创建并维护全部标准化 Jinja 宏（sap_date / sap_string / sap_flag / audit_columns 等）。
依赖：无（全局触发）  →  产出：10 个标准宏 + 编码规范手册
01
AGENT-01
场景分析
负责：解析业务报表原型，识别维度、度量、筛选条件。逆向设计五层全链路模型，输出字段级映射表和 ETL 路径规划。
依赖：Agent-00  →  产出：全链路模型设计文档 + 字段映射表
02
AGENT-02
ODS 搭建
负责：连接源系统读取元数据，自动生成 sources.yml（含字段注释和 freshness 监控）。ODS 层只读，不修改源数据。
依赖：Agent-01  →  产出：sources.yml（所有源表定义）
03
AGENT-03
DWD 开发
负责：1:1 清洗每张 ODS 表，自动完成类型转换、NULL 默认值、ROW_NUMBER 去重、7 列审计字段注入。禁止跨表 JOIN，保留所有行。
依赖：Agent-02  →  产出：DWD 清洗模型 + 域级 schema.yml
▼ ▼ ▼
04
AGENT-04
DWS 开发
负责：多表 JOIN 和业务计算——发票基表→金税桥接→数据分流→维度补全→定价成本→财务计划→业务宽表。按 ETL 链路分步生成中间模型。
依赖：Agent-03  →  产出：DWS 中间模型 + ETL 链路
05
AGENT-05
DIM 设计
负责：从 DWD 主数据域提取维度，实现客户维度（SCD Type 2）、产品维度（SCD Type 1），生成代理键。维度表被 DWS 和 ADS 共同引用。
依赖：Agent-03  →  产出：共享维度模型 + SCD 策略
06
AGENT-06
ADS 开发
负责：消费 DWS 宽表和 DIM 维度，生成 BI 就绪参数化视图。支持多值筛选、日期范围、联动过滤。不在 ADS 做 JOIN，实现 Hive + OceanBase 双写。
依赖：Agent-04 + Agent-05  →  产出：ADS 参数化视图 + BI 数据集
07
AGENT-07
迁移验证
负责：8 步迁移流程（DDL→全量→增量→行数→金额→主键→抽样→BI 连接），自动生成四维一致性验证报告。验证报告不得隐瞒差异。
依赖：Agent-06  →  产出：迁移脚本 + 四维一致性报告

◇ AI 协同模式

从 Agent 定义自动构建 System Prompt，注入 project.yml 项目上下文（名称、业务域、分层策略、连接信息）。Agent 之间通过依赖拓扑自动排序，输出可执行的 prompt 文本。

◇ 直接执行模式

直接执行 dbt 子进程命令。根据 Agent 的层权限和工具链自动构建 dbt run / test / parse 命令序列。支持 dry-run 预览、失败续跑、超时控制。

运行逻辑

平台是如何运转的

从一份 project.yml 配置文件开始，到 BI 看板交付结束——8 个环节端到端自动化

1

项目配置

在 project.yml 中定义项目名、源系统适配器、业务域、数据分层和场景编排。引擎层所有代码读取这份配置，零硬编码。

2

元数据发现

适配器连接源系统，读取所有源表的表名、列名、数据类型、主键和注释，生成 sources.yml 和字段盘点清单。

3

模型生成

SQL 引擎读取字段清单，按层生成标准化 dbt 模型。DWD 自动完成类型转换和去重，DWS 按 ETL 链路分步生成。

4

本地验证

模型先在本地 DuckDB 上 dbt run --target dev，秒级验证编译和运行正确性，避免在生产集群反复试错。

5

生产执行

dbt-core DAG 引擎自动解析 ref() 依赖，按拓扑顺序在 Hive Spark 集群上执行。支持全量 + 增量两种物化策略。

6

跨库验证

数据写入 OceanBase 后，自动对比行数、金额、主键、抽样数据。差异报告 JSON 输出，可集成到 CI/CD 流程。

7

看板交付

BI 引擎读取 YAML 配置，查询 ADS 模型，经 ECharts 渲染为完整的 HTML 交互看板。一个命令批量生成全部看板。

8

定时调度

Git push 触发 CI 自动门禁。DolphinScheduler 按场景 cron 定时执行。失败自动告警，全链路可观测。

日常开发流程

# 1. Agent 在本地 DuckDB 生成并验证模型 dbt-platform agent pipeline --target dev
# 2. 检查 Agent 产出（可选） dbt-platform agent status dbt-platform scan --json
# 3. 确认无误，推到 Hive 生产 dbt-platform agent pipeline --target hive dbt-platform validate
